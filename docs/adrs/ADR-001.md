# ADR-001: Session Type Selector Modal Architecture

> **Status:** accepted
> **Deciders:** Claude (Sonnet 4.5), Development Team
> **Date:** 2026-02-22
> **PRD:** [PRD-001](../prds/PRD-001.md)

---

## Context

Users need to choose between creating a Claude Code session or a regular terminal session when pressing Ctrl+T or clicking "New Session". We need to decide on the architecture for implementing this session type selector.

Key considerations:

- Must integrate with existing Zustand state management
- Must use shadcn/ui components for consistency
- Must not break existing keyboard shortcuts
- Must be performant (modal render < 100ms)
- Must be accessible (WCAG 2.1 AA)

---

## Decision Drivers

- Team expertise in React, Zustand, and shadcn/ui
- Existing design system (Catppuccin Mocha)
- Performance requirements (< 100ms modal render)
- Consistency with existing dialog implementations
- Accessibility requirements (keyboard navigation, screen readers)

---

## Considered Options

### Option 1: New Dedicated Modal Component with Dedicated Store

**Description:** Create a new `NewSessionDialog` component with its own dedicated Zustand store (`useNewSessionDialogStore`) for managing open/close state.

**Pros:**

- Clear separation of concerns
- Easy to test in isolation
- Explicit state management
- Follows existing pattern (like `command-palette.ts`)

**Cons:**

- Adds another store (might be overkill for simple boolean state)
- More boilerplate code

**Estimated Effort:** Low

---

### Option 2: Extend Existing `useAppDialogsStore`

**Description:** Add `newSessionOpen` state to the existing `useAppDialogsStore` and create a new `NewSessionDialog` component.

**Pros:**

- Reuses existing infrastructure
- Less code overall
- Consistent with existing dialogs (shortcuts, about)
- Simpler mental model

**Cons:**

- Couples session logic to general app dialogs
- Store could become bloated over time

**Estimated Effort:** Low

---

### Option 3: Inline Modal in Command Palette

**Description:** Embed the session type selector directly in the Command Palette as a multi-step flow.

**Pros:**

- No new components needed
- Familiar UI (users already know command palette)

**Cons:**

- Breaks command palette mental model (search vs. selection)
- Less visual clarity for session type choice
- Harder to trigger from keyboard shortcut
- Complicates command palette logic

**Estimated Effort:** Medium

---

## Decision

**Chosen Option:** Option 2 - Extend Existing `useAppDialogsStore`

**Rationale:**

1. **Consistency:** The `useAppDialogsStore` already manages dialog state for `KeyboardShortcutsDialog` and `AboutDialog`. Adding `newSessionOpen` follows the established pattern.

2. **Simplicity:** We don't need the complexity of a dedicated store for a single boolean flag. The app dialogs store is designed for exactly this use case.

3. **Low Effort:** Minimal code changes required. We just add one property and two methods to the existing store.

4. **Maintainability:** Keeps all dialog state in one place, making it easier to debug and test.

5. **Performance:** No performance difference between Option 1 and Option 2.

---

## Consequences

### Positive

- Consistent state management pattern across all dialogs
- Less code to maintain
- Easy to extend for future dialogs
- Familiar pattern for developers

### Negative

- `useAppDialogsStore` grows slightly (but remains manageable)
- If we add many more dialogs, we may need to refactor later (but not a concern yet)

### Risks

- **Risk:** Store becomes bloated with too many dialogs
  - **Mitigation:** Monitor store size; refactor if it exceeds 5-7 dialogs

---

## Technical Details

### Architecture

```
App.tsx
  └── NewSessionDialog (component)
      ├── uses useAppDialogsStore (newSessionOpen state)
      ├── uses useTerminalTabsStore (addTab, nextTabId)
      ├── uses useFileTreeStore (currentPath)
      └── shadcn/ui Dialog (primitive)
```

### Key Components

| Component | Responsibility | Technology |
|-----------|----------------|------------|
| `NewSessionDialog` | Render modal, handle user selection | React 19 + TypeScript |
| `useAppDialogsStore` | Manage modal open/close state | Zustand |
| `useTerminalTabsStore` | Create new tabs with session type | Zustand |
| shadcn/ui Dialog | Modal primitive (overlay, focus trap) | Radix UI |

### State Changes

**Before:**

```typescript
interface AppDialogsState {
  shortcutsOpen: boolean;
  aboutOpen: boolean;
  setShortcutsOpen: (open: boolean) => void;
  setAboutOpen: (open: boolean) => void;
}
```

**After:**

```typescript
interface AppDialogsState {
  shortcutsOpen: boolean;
  aboutOpen: boolean;
  newSessionOpen: boolean;
  setShortcutsOpen: (open: boolean) => void;
  setAboutOpen: (open: boolean) => void;
  setNewSessionOpen: (open: boolean) => void;
}
```

### Integration Points

- **Keyboard Shortcuts:** Intercept Ctrl+T in `App.tsx`, call `setNewSessionOpen(true)`
- **Command Palette:** Update "New Session" command to call `setNewSessionOpen(true)`
- **Tab Creation:** Call `addTab(tabId, path, sessionType)` when user selects type

## Implementation Notes

### Prerequisites

- [ ] Verify Tauri backend supports session type parameter
- [ ] Confirm `useTerminalTabsStore` can accept session type

### Migration Strategy

Not applicable (new feature, no migration needed).

### Rollback Plan

If issues arise:

1. Remove `NewSessionDialog` component
2. Remove `newSessionOpen` from `useAppDialogsStore`
3. Restore direct tab creation in keyboard shortcut handler
4. Restore direct tab creation in command palette
