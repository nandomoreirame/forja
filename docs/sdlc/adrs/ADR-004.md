# ADR-004: Workspace Color & Icon â€” Extend Existing Model, Inline Dropdown Editor

**Status:** Accepted
**Date:** 2026-03-15
**Deciders:** Fernando Moreira (nandomoreira)

---

## Context

The existing `Workspace` data model in `electron/config.ts` only stores `id`, `name`,
`projects`, `createdAt`, and `lastUsedAt`. The UI only offered a plain "Create Workspace"
dialog. The user wants rich visual identity per workspace and a first-class topbar
switcher, referencing four design screenshots.

## Decision

### 1. Extend the `Workspace` interface â€” not replace it

Add `color: WorkspaceColor` and `icon: WorkspaceIcon` as optional fields with
backward-compatible defaults (`"mauve"` / `"layers"`). This avoids a migration
and keeps existing workspaces functional.

```typescript
// electron/config.ts (authoritative)
export interface Workspace {
  id: string;
  name: string;
  color?: WorkspaceColor;  // default "mauve"
  icon?: WorkspaceIcon;    // default "layers"
  projects: string[];
  createdAt: string;
  lastUsedAt: string;
}
```

The frontend store mirrors this with the same type, resolving defaults at render-time.

### 2. Dropdown as a single custom component (`WorkspaceSwitcher`)

We build a bespoke `<WorkspaceSwitcher>` component rather than adapting
`DropdownMenu` + `Dialog` combo. Rationale:

- The inline edit mode (expanded row) cannot be expressed in standard shadcn/ui
  `DropdownMenu` items without fighting the close-on-click behavior.
- A custom popover controlled by `useState(open)` gives us full control over
  when the dropdown closes (e.g., stay open while editing a workspace).

Implementation uses `@radix-ui/react-popover` directly (already available via shadcn
peer deps) wrapped in a `<WorkspaceSwitcher>` component.

### 3. IPC `update_workspace` extended to accept color + icon

The existing `update_workspace` IPC handler only accepted `{ id, name }`. We extend
its signature:

```typescript
ipcMain.handle("update_workspace", async (_event, args: {
  id: string;
  name?: string;
  color?: string;
  icon?: string;
}) => { ... });
```

The `updateWorkspace` function in `config.ts` already uses `Partial<Pick<Workspace, "name">>`;
we extend the Pick to include `color` and `icon`.

### 4. New window for "+ Create new workspace"

Clicking "+ Create new workspace" calls the existing `open_workspace_in_new_window`
IPC channel (already implemented in `main.ts`), but with `undefined` workspaceId so
the new window starts fresh. The current window is completely untouched â€” no sessions
are killed.

### 5. Icon rendering via Lucide React

Icons are rendered using Lucide React components. The mapping of
`WorkspaceIcon` string â†’ Lucide component lives in a single
`frontend/lib/workspace-icons.ts` lookup table, keeping component imports
tree-shakable and the icon set easily extensible.

---

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|-----------------|
| Store color as hex string | String is less type-safe; enum/union ensures only valid colors are stored |
| Use shadcn `DropdownMenu` for switcher | Close-on-click behavior makes inline edit mode impossible without hacks |
| Separate "edit workspace" dialog | Worse UX than inline edit per the design screenshots |
| Kill sessions on workspace switch | Defeats the purpose of multi-workspace; sessions are precious |

---

## Consequences

- `electron/config.ts` `Workspace` interface gains two optional fields
- `frontend/stores/workspace.ts` `Workspace` interface gains the same two fields
- `frontend/components/workspace-switcher.tsx` is a new component (~200â€“300 lines)
- `frontend/lib/workspace-icons.ts` is a new utility module
- `electron/main.ts` `update_workspace` handler signature is extended
- `electron/config.ts` `updateWorkspace()` accepts `color` and `icon`
- Existing workspaces without color/icon silently default to mauve/layers
