# PRD-004: Workspace System — Rich Workspaces with Color, Icon & Multi-Window

**Status:** Draft
**Date:** 2026-03-15
**Author:** Claude (sdlc-workflow)
**Feature Branch:** `feature/tiling-layout` (extend existing branch)

---

## 1. Overview

Forja currently stores a basic `Workspace` record (id, name, projects, dates) in
`electron-store`. The UI has no first-class workspace switcher: workspaces are only
accessible through the project-sidebar's context-menu and a plain dialog.

This PRD defines the next evolution: **rich workspaces** with visual identity (color +
icon), a **topbar dropdown switcher**, and a **+ Create new workspace** flow that opens
a new Electron window while keeping every other workspace's PTY sessions alive.

---

## 2. Goals

| # | Goal |
|---|------|
| G1 | Users can see the active workspace in the topbar at a glance (icon + name) |
| G2 | Users can switch between workspaces from a dropdown without killing other sessions |
| G3 | Users can give each workspace a name, color, and icon |
| G4 | Users can create a new workspace in a new window while keeping active sessions live |
| G5 | Users can delete a workspace (with confirmation) |

## 3. Non-Goals

- No cloud sync of workspace state
- No workspace sharing / collaboration
- No workspace import/export in this iteration
- Sessions are NOT automatically killed when switching workspaces (they stay alive)

---

## 4. User Stories

- **US-004** — As a developer, I want a topbar workspace switcher so I can quickly see
  which workspace is active and switch to another without losing my sessions.

## 5. Feature Details

### 5.1 Workspace Data Model (extended)

```typescript
export interface Workspace {
  id: string;
  name: string;
  color: WorkspaceColor;   // NEW: one of 7 Catppuccin colors
  icon: WorkspaceIcon;     // NEW: one of 14 named icons
  projects: string[];
  createdAt: string;
  lastUsedAt: string;
}

export type WorkspaceColor =
  | "green"   // #a6e3a1
  | "teal"    // #94e2d5
  | "blue"    // #89b4fa
  | "mauve"   // #cba6f7
  | "red"     // #f38ba8
  | "peach"   // #fab387
  | "yellow"; // #f9e2af

export type WorkspaceIcon =
  | "waves" | "mountain" | "star" | "heart" | "bolt" | "cloud"
  | "moon"  | "layers"   | "rocket" | "beaker" | "link" | "trending"
  | "graduation" | "coffee";
```

### 5.2 Topbar Workspace Button

- Positioned **to the right of the Menu button** in the left cluster of the Titlebar
- Shows: `[colored-icon]` — compact, no text unless hovered
- On hover tooltip: workspace name
- On click: opens the **Switch Workspace dropdown**

### 5.3 Switch Workspace Dropdown

Title: "Switch workspace"

For **each workspace** in the list:

- Row: `[icon]  [name]`  (inactive)
- Active workspace row is expanded **inline** into edit mode showing:
  - Name input (editable, auto-focused on edit click)
  - Color picker: 7 circles (green, teal, blue, mauve, red, peach, yellow)
  - Icon picker: 14 icon buttons in a 7×2 grid
  - Edit (pencil) / Confirm (checkmark) toggle buttons
  - "Delete workspace" red link
- Footer: `[+] Create new workspace` button

### 5.4 Create New Workspace Flow

1. User clicks "+ Create new workspace"
2. The **current window's workspace state is saved** (projects, sessions are untouched)
3. A new Electron BrowserWindow opens (via `open_workspace_in_new_window`)
4. New window starts with a blank workspace that the user names inline
5. Previous window stays fully alive — all PTY sessions continue running

### 5.5 Session Preservation Policy

- Switching workspaces via the dropdown does **NOT** kill PTY sessions in other
  workspaces.
- Sessions are only killed when the user explicitly closes them (existing behavior).
- This means multiple Electron windows can be open simultaneously, each with their
  own workspace, and all sessions remain alive.

Screenshots show:

- The workspace icon appears as a small colored icon in the topbar left cluster

## 7. Acceptance Criteria

| ID | Criterion |
|----|-----------|
| AC-1 | Workspace button visible in topbar (left cluster, after Menu) |
| AC-2 | Dropdown lists all workspaces; active workspace is highlighted |
| AC-3 | Active workspace supports inline name/color/icon editing |
| AC-4 | Color change is reflected immediately on the topbar button |
| AC-5 | Icon change is reflected immediately on the topbar button |
| AC-6 | "Delete workspace" shows confirmation, then removes workspace |
| AC-7 | "+ Create new workspace" opens a new window, current window unaffected |
| AC-8 | Clicking an inactive workspace switches to it (sets activeWorkspaceId) |
| AC-9 | Data (color + icon) persists after app restart |
| AC-10 | All new fields have TypeScript types; no `any` |
| AC-11 | Tests cover store logic, IPC handlers, and component rendering |

---

## 8. Technical Constraints

- Must be backward-compatible: existing `Workspace` records without `color`/`icon`
  must default to `"mauve"` and `"layers"` respectively.
- IPC handlers for `update_workspace` must accept the new fields.
- The `Workspace` interface in `electron/config.ts` and `frontend/stores/workspace.ts`
  must stay in sync.

## 9. Out of Scope

- Workspace-specific settings (theme, fonts) (future)
